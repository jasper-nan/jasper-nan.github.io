RAG 检索增强生成实战：让 LLM 读懂你的私有文档

从零搭建 RAG 系统：文档切片、向量检索、重排序，让大模型拥有你的私有知识

Wednesday, May 7, 2026

一句话理解 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）就是给大模型配一个"外接硬盘"。模型本身记不住你的私有文档、公司内部资料、最新产品手册，但 RAG 能先把这些文档切片存进向量数据库，当你提问时，先检索最相关的片段，再让模型基于这些片段回答。

简单说：RAG = 文档检索 + 上下文注入 + LLM 生成

RAG 解决了什么问题

大模型有三个天然短板：

  • 知识截止日期：模型训练数据有截止时间，对最新事件一无所知
  • 没有私有数据：你的公司内部文档、产品手册、客户记录，模型从来没见过
  • 幻觉问题：模型会一本正经地编造不存在的信息

RAG 的解决思路很直接：不依赖模型记住所有知识，而是让它按需查阅资料。就像开卷考试——学生不需要背下整本教科书，只要会快速翻到相关章节就行。

RAG 系统架构拆解

1. 文档加载与切片（Ingestion）

第一步是把各种格式的文档读进来，切成模型能消化的小块。

# 用 LangChain 加载 PDF
from langchain.document_loaders import PyPDFLoader

loader = PyPDFLoader("产品手册.pdf")
pages = loader.load()  # 每页一个 Document 对象

# 切片：按字符数切，保留上下文重叠
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # 每块 500 字符
    chunk_overlap=50,    # 重叠 50 字符，防止断句
    separators=["\n\n", "\n", "。", " ", ""]  # 优先按段落切
)
chunks = splitter.split_documents(pages)
print(f"切成 {len(chunks)} 块")

切片策略很关键：

  • Chunk 太大：检索精度下降，无关内容多
  • Chunk 太小：上下文断裂，模型理解不了
  • 一般 300-800 字符为宜，技术文档可以按标题层级切

2. 向量化与存储（Embedding + Vector DB）

把文本块变成向量（高维数字数组），存入向量数据库。相似语义的文本在向量空间里距离更近。

# 使用 OpenAI 的 Embedding API
from langchain.embeddings import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 存入向量数据库（以 Chroma 为例）
from langchain.vectorstores import Chroma

vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"  # 持久化到本地
)

向量数据库选型：

数据库特点适用场景
Chroma轻量、本地优先、开箱即用原型开发、个人项目
Milvus分布式、高性能、云原生企业级生产环境
QdrantRust 编写、过滤查询强需要元数据过滤的场景
WeaviateGraphQL 接口、模块化需要复杂查询语法

3. 检索（Retrieval）

用户提问时，先把问题向量化，然后在向量库里找最相似的文档块。

# 相似度检索：返回 top-k 最相关的块
query = "产品的退款政策是什么？"
retrieved_chunks = vectorstore.similarity_search(query, k=5)

for chunk in retrieved_chunks:
    print(f"相似度分数: {chunk.metadata.get('score')}")
    print(f"内容: {chunk.page_content[:200]}...")
    print("---")

检索优化技巧：

  • MMR（最大边际相关性）：在相关性和多样性之间平衡，避免返回内容重复的块
  • 元数据过滤：先按文件名、日期、标签过滤，再向量检索
  • 混合检索：向量相似度 + 关键词匹配（BM25）结合，召回率更高

4. 重排序（Re-ranking）

向量检索速度快但精度有限。用更重的模型对初步结果二次排序，能显著提升相关性。

# 使用 Cross-Encoder 重排序
from sentence_transformers import CrossEncoder

reranker = CrossEncoder('BAAI/bge-reranker-base')

# 对检索结果重排
pairs = [[query, chunk.page_content] for chunk in retrieved_chunks]
scores = reranker.predict(pairs)

# 按重排分数排序
reranked = sorted(
    zip(retrieved_chunks, scores),
    key=lambda x: x[1],
    reverse=True
)

# 只保留前 3 个最相关的
final_chunks = [chunk for chunk, score in reranked[:3]]

重排序模型（如 bge-reranker、cohere-rerank）专门训练来判断"问题和文档的相关性"，比通用 Embedding 模型更精准。

5. 生成回答（Generation）

把检索到的文档块拼接成上下文，塞进 Prompt，让模型回答。

from langchain.chains import RetrievalQA
from langchain.chat_models import ChatOpenAI

llm = ChatOpenAI(model="gpt-4", temperature=0)

# 构建 RAG Chain
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",  # 把所有文档块塞进一个 prompt
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
    return_source_documents=True  # 返回引用的文档来源
)

# 提问
result = qa_chain({"query": "退款需要多久到账？"})
print(result["result"])
print("\n引用来源:")
for doc in result["source_documents"]:
    print(f"- {doc.metadata.get('source', '未知')}")

完整代码示例

一个最小可用的 RAG 系统，不到 50 行：

import os
from langchain.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA

# 1. 加载文档
loader = TextLoader("faq.txt")
docs = loader.load()

# 2. 切片
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

# 3. 存入向量库
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(chunks, embeddings)

# 4. 构建问答链
llm = ChatOpenAI(temperature=0)
qa = RetrievalQA.from_chain_type(llm=llm, retriever=vectorstore.as_retriever())

# 5. 提问
print(qa.run("如何申请退款？"))

进阶优化方向

优化点方法效果
查询改写HyDE（假设文档嵌入）、查询扩展提升检索召回率
多路召回向量检索 + BM25 + 图谱检索覆盖不同查询类型
上下文压缩LLMLingua、选择性上下文减少 Token 消耗，提升精度
自适应切片按语义边界切（而非固定长度）保持段落完整性
Agentic RAG让 Agent 决定是否需要检索、检索什么复杂多步推理场景

写在最后

RAG 不是银弹，但它是让 LLM 落地企业场景最有效的路径。不需要微调模型，不需要准备大量标注数据，只要把文档整理好、切片合理、检索精准，就能让一个通用大模型变成你的"领域专家"。

如果你刚开始接触 RAG，建议先用 LangChain + Chroma 搭一个最小原型，跑通自己的文档。优化是第二步，先让系统转起来，比追求完美架构更重要。
