Tuesday, March 31, 2026

OpenClaw 好玩的地方，不是它“能装很多 Skill”，而是你能不能把它调成一个真正顺手的个人系统。最近正好做了一轮整理：砍掉重复能力、收敛搜索入口、补上健康检查和安全配置。折腾完之后，一个很明显的感受是——Agent 不是插件市场，Agent 应该是你的工作流延伸 。

一、先说结论：OpenClaw 不是装得越多越好

很多人刚接触 OpenClaw，第一反应都是：

  • 先装一堆 Skill 再说
  • 搜索、浏览器、平台抓取、自动化各来一套
  • 能接的消息平台全部接上

刚开始这样玩很爽，但很快就会遇到几个问题：

  1. 同一类功能入口太多 ，模型容易摇摆
  2. Skill 互相重叠 ，实际价值并没有叠加
  3. 维护成本上升 ，更新、兼容、排错都更麻烦
  4. 安全面变大 ，尤其是接了群聊和外部触发后

所以我这次优化的核心思路很简单：

一个场景只留一个主入口；平台型能力按需补充；能禁用的重复能力就禁用。

二、我最后保留了哪些 Skill

1. multi-search-engine：默认网页搜索入口

这是我最后保留下来的默认搜索 Skill。

原因很直接：

  • 支持多个搜索引擎
  • 适合做通用网页检索
  • 没有太多平台耦合
  • 适合“先搜、再读、再总结”的标准链路

我现在的策略是：

  • 一般网页搜索 → multi-search-engine
  • 只把它当成默认搜索入口

这件事看起来只是一个偏好问题，实际上会直接影响 Agent 的行为稳定性。默认入口越清晰，模型越不容易在多个工具之间乱跳。

2. agent-reach：平台型内容搜索

如果说 multi-search-engine 负责搜网页，那 agent-reach 更适合搜平台：

  • X / Twitter
  • Reddit
  • GitHub
  • B 站
  • 小红书
  • 播客 / RSS / 一些垂直内容源

所以我的分工是：

  • 默认搜索：multi-search-engine
  • 平台型搜索：agent-reach

这样做的好处是，模型遇到“帮我搜一下某个网站/平台内容”时，有一个很明确的次级入口，不至于什么都走同一种搜索。

3. caldav-sync：把日历接入个人系统

这个 Skill 的价值不在于炫技，而在于它足够贴近生活流。

一旦你本来就有自己的 CalDAV 服务，Agent 直接读写日历会非常自然：

  • 查今天 / 明天日程
  • 生成提醒
  • 做简单的日程协同

它不是那种“一眼惊艳”的能力，但属于接入之后就很难再拿掉的基础能力。

4. money-accounting：和真实生活闭环的 Skill

很多 Skill 只是“技术上能做”，但不一定真的高频。记账类 Skill 则不一样，它很容易直接进入日常动作里。

比如：

  • 截图识别账单
  • 快速记录支出
  • 和自己的记账系统打通

这种 Skill 一旦和你原本就存在的行为链路接起来，价值会非常高。

5. garmin-connect：强垂直、高匹配时很有价值

如果你有 Garmin 设备，这个 Skill 就非常合适：

  • 跑步记录
  • 睡眠数据
  • 心率 / 步数 / HRV
  • 近期训练汇总

但这类 Skill 的原则也很简单：高匹配就保留，不匹配就不要因为“看起来强”而硬装。

6. server-health：把系统状态做成一个随时可查的简报

这是我最近新加的一项能力，也是很值得长期保留的一个方向。

它能快速查看：

  • CPU / 内存 / 磁盘 / uptime
  • Top 进程
  • OpenClaw Gateway 状态
  • Docker 容器情况

原始版本更偏 Linux，我自己又做了一轮 macOS 兼容适配。对有 NAS、Mac mini、自建服务的人来说，这一类 Skill 非常适合长期留着。

三、我主动禁用了哪些重复能力

1. 禁用系统内置 clawhub skill

原因不是它不好，而是我不希望“找 Skill / 装 Skill”这件事在日常对话里过于主动地出现。

对我来说，安装 Skill 应该是一个显式操作，不应该成为日常对话默认的分支路径。

2. 禁用系统内置 web search

这是这次优化里我觉得最值的一步。

如果系统里同时存在：

  • 内置 web search
  • 第三方搜索 Skill
  • 平台搜索 Skill

模型就很容易“看心情走工具”。结果就是：有时行为好，有时行为怪，复现性很差。

所以我直接做了收口：

  • 内置 web search 关掉
  • 默认搜索统一走 multi-search-engine
  • 平台搜索统一走 agent-reach

这一步的收益非常大，属于典型的“看起来像做减法，实际上是在提稳定性”。

3. 谨慎对待浏览器类重复能力

OpenClaw 本身已经有浏览器能力，如果再叠加一层功能接近的外部 Skill，很多时候只是让决策路径变复杂。

我的结论是：

同类能力尽量只保留一套主入口。

四、除了 Skill，本体配置也值得优化

折腾了一圈之后我越来越觉得，OpenClaw 真正值得优化的，不只是 Skill 列表，配置本身更重要 。

1. 清理旧会话

如果长期运行，OpenClaw 的 session 会逐渐堆积，尤其是 cron / heartbeat / 历史通道会话。适当清理有几个好处：

  • 减少噪音
  • 降低维护成本
  • 让诊断信息更清爽

2. 显式配置 plugins.allow

如果你有自定义插件，最好明确写 trusted allowlist，而不是让系统自动发现后直接加载。这样更可控，也更安全。

3. 精简模型 fallback 链

模型回退链不是越长越好。

链太长会有两个问题：

  • 行为风格不一致
  • 排查问题时很难确认到底是谁在回答

我最后保留的是一个更简洁的链路：

  • 主模型：gpt-5.4
  • Fallback 1：glm-5-turbo
  • Fallback 2：MiniMax-M2.7

这样既保留可用性，也不会让链路变得太难理解。

五、我现在的实践原则

经过这轮收敛之后，我给自己定了几个很简单但很有效的规则：

规则 1：一个场景只保留一个默认入口

  • 网页搜索 → multi-search-engine
  • 平台搜索 → agent-reach

规则 2：高频优先，低频能删就删

优先保留：

  • 搜索
  • 日历
  • 记账
  • 健康检查
  • 真正进入日常流的垂直 Skill

规则 3：先收敛，再扩展

比起不停安装新的 Skill，我现在更倾向于：

  • 先把已有能力整理顺
  • 让默认路径稳定下来
  • 确定缺口后再补技能

一个 10 分但顺手的系统，远比一个 100 分但混乱的系统更适合长期使用。

六、如果你也在搭 OpenClaw，我的建议是什么

如果你也准备折腾自己的 OpenClaw，我建议先从一个最小可用组合开始：

  1. 一个默认搜索 Skill
  2. 一个平台型搜索 Skill
  3. 一个日历 / 提醒能力
  4. 一个健康检查 / 运维简报能力
  5. 一个与你生活强相关的垂直 Skill（比如记账、运动、知识库）

只要这几个打通，体验通常已经很好了。

剩下的，不急着装，先看自己是不是会高频用到。

七、结语

我越来越觉得，OpenClaw 最有意思的地方，不在于它“功能很多”，而在于：

你能不能把它调成一个适合你自己的系统。

不是越全越好，而是越顺手越好。不是能做多少事，而是能不能真的进入你的工作流。

对我来说，这轮优化最大的收获不是多了几个 Skill，而是把这件事彻底想明白了：

Agent 不是软件商店，Agent 是你的操作习惯的延伸。

如果你也在折腾 OpenClaw，不妨先做一次“减法优化”。很多时候，真正提升体验的不是再装一个 Skill，而是把入口收口、把边界理清、把高频路径调顺。
