AI 助手实战：我用 OpenClaw 打造个人自动化系统

一台 Mac mini + 十个自动化场景，帮我省下每天两小时

Sunday, March 29, 2026 6 minutes read

凌晨两点，Mac mini 的风扇安静地转着，客厅的小机器正在替我操心一堆杂事：检查基金净值、整理今天的步数、给我推送一篇值得读的周报文章。这事说起来挺魔幻的——大概半年前我还在手动复制粘贴各种数据，现在这些全自动化了。

这篇文章想跟你聊聊我是怎么用 OpenClaw 搭建这套系统的，中间踩了哪些坑，以及它现在到底帮我省了多少时间。

先说说背景：我是谁，我想要什么

我叫 Jasper，32 岁，在武汉一家物流科技公司做全栈开发。原来是测试工程师出身，写了几年自动化脚本后来转了开发，对"让机器做重复的事"这件事有天然的执念。

坐标武汉，家里有一台 Mac mini（M4 Pro 那款，部署 OpenClaw 主力机），两台 NAS（一台 8 盘位飞牛 + 一台 4 盘位微联通），还有一块 Garmin Fenix 7。

我的核心需求很简单：工作之外的事情，能自动化就自动化，别让我操心。

具体来说，我想实现这些：

  1. 每天早上 8 点，收到一份简报——今天有什么待办、基金净值怎样、有没有值得看的文章
  2. 记账不需要手动打开 App，截图就能自动识别写入
  3. 能随时问 AI 助手问题，搜资料，不需要自己开浏览器
  4. 运动数据（跑步、骑行、睡眠）能随时调出来
  5. 家里 NAS 里的文件能方便管理，不用每次都 RDP 过去

为什么选 OpenClaw

在动手之前我也调研了一圈，最终选 OpenClaw 主要有几个原因。

第一个原因是它的通道支持非常全面。 我同时用飞书、QQ、微信，之前的方案（比如用企业微信机器人或者单独搭 QQ Bot）每个都要单独部署，维护成本很高。OpenClaw 一个系统把几个通道都接入了，我可以统一管理，这是最吸引我的点。

第二个原因是它的 Skill 生态。 OpenClaw 有 Skills 市场，很多常见需求（天气、提醒、飞书文档、Google 日历等）已经有现成的封装，拿来就能用，不需要从零写 prompt 工程。

第三个原因是它跑在本地。 我的 Mac mini 7x24 小时开着，数据不经过第三方服务器，隐私相对可控。对于记账、运动数据这些私人信息，这个很重要。

第四个原因是开发者友好。 OpenClaw 基于 Node.js，配置文件是 YAML/JSON，Python 写 Skill 也完全没问题。我这种会写点代码的人，改起来不吃力。

当然它也有一些局限性——比如有些 Skill 更新不够频繁，部分功能依赖特定的 API 接口，但总体来说，用它搭建这套系统的性价比是最高的。

部署过程：从零到跑起来用了多久

说实话，比我预期的快。

第一步：安装

我是用 Docker 装的，官方文档写得很清楚，一行命令拉起来：

docker run -d \
  --name openclaw \
  --restart unless-stopped \
  -p 3000:3000 \
  -v ~/openclaw/config:/app/config \
  -v ~/openclaw/data:/app/data \
  openclaw/openclaw:latest

装完之后打开 http://localhost:3000，第一次需要初始化管理员账号，这个跟着向导走就行，没什么坑。

第二步：配置通道

这一步是重头戏。我先后接入了三个通道：

飞书：需要先在飞书开放平台创建一个企业自建应用，拿到 App ID 和 App Secret，然后在 OpenClaw 的配置里填进去。消息加密那块要注意，飞书用的是 AES 加解密，密钥要配对，不然收不到消息。

QQ 频道：需要申请 QQ 机器人权限，在 QQ 开放平台创建应用，然后把 Token 配置进 OpenClaw。QQ 的限制比较多，比如每分钟消息条数有上限，这个后面再说。

微信通道：通过 WeChat Work（企业微信）接入，相对来说流程最顺畅，接入稳定性也最高。

第三步：配置定时任务（Cron）

OpenClaw 的定时任务配置很灵活，直接在配置文件里写 Cron 表达式：

cron:
  - name: morning-brief
    schedule: "0 8 * * 1-5"
    task: skills/morning-brief
  - name: fund-value
    schedule: "0 9 * * *"
    task: skills/fund-value
  - name: weekly-reading
    schedule: "0 10 * * 5"
    task: skills/weekly-reading

第一次配 Cron 的时候容易搞错时区，OpenClaw 默认用 UTC，我配置的是 Asia/Shanghai，这个细节错了就会导致任务执行时间差 8 小时。一定要检查时区设置。

第四步：接入外部服务

这步花的时间最长，因为每个服务的 API 和认证方式都不一样。

CalDAV 日历：我用一个自建的 CalDAV 服务（caldav.laji.fun），OpenClaw 通过 CardDAV/CalDAV 协议读取日历数据，然后用来做待办提醒和日程管理。

记账系统：我的记账用的是 money.laji.fun 这个自建项目，截图 OCR 用了本地部署的 Qwen3.5-27B 模型（跑在另一台机器上，IP 192.168.9.245:9990），识别准确率相当高。具体流程是：截图 → 通过 OpenClaw Skill 调用 OCR API → 结构化数据写入 money.laji.fun。

NAS 文件管理：两台 NAS 都通过 SFTP 访问，飞牛的 IP 是内网地址，通过 Tailscale 打通了内网穿透，OpenClaw 直接用 SFTP 协议操作文件。

整个部署过程花了大概一个周末，其中踩坑主要在飞书消息加解密和 CalDAV 认证这两个地方。

实际使用场景：我是怎么用它的

场景一：每天早间简报

每天早上 8 点（工作日），OpenClaw 自动给我推送一条飞书消息，内容包括：

  • 今日日历（从 CalDAV 读出来的）
  • 基金净值（手动配置的几个基金代码，通过爬虫抓取）
  • 今日待办（从 Things 3 读取）
  • 一条随机推送（可能是阅读周报、天气提醒，或者我自定义的提示语）

这个简报省去了我每天打开好几个 App 刷一遍的步骤，早上刷牙的时候看一眼飞书，就知道今天大概要干什么了。

实现这个功能用到了一个核心 Skill：morning-brief，里面串了日历读取、基金爬虫、Things 3 接口三个数据源，然后拼成一条格式化的消息。

场景二：截图记账

这是我用得最频繁的功能之一。

以前记账是这样的：打开记账 App → 手动输入金额、分类、备注 → 保存。平均一笔大概 1 分钟。

现在是：截图 → 发给 OpenClaw → 自动识别写入记账系统。

具体流程：

  1. 微信/QQ 发一张支付截图给 OpenClaw
  2. OpenClaw 调用本地 OCR API（Qwen3.5-27B）识别金额、商家、分类
  3. 写入 money.laji.fun，同时回复确认消息

OCR 识别的准确率大概在 90% 左右，金额基本不会错，分类偶尔会分错（比如把"超市"分到"餐饮"而不是"日用"），但发一条消息纠正就行，整体效率提升非常明显。

场景三：多搜索引擎信息搜索

OpenClaw 集成了一套多搜索引擎系统，支持百度、Google、DuckDuckGo 等 17 个搜索引擎。我配置了一个快捷指令：发一条消息带上关键词，OpenClaw 自动在多个引擎并行搜索，然后汇总结果去重后发给我。

这个在研究技术方案、对比产品评测的时候特别有用，不用一个一个引擎手动搜了。

场景四：Garmin 运动数据查看

跑步和骑行是我的主要运动。以前看运动数据要打开 Garmin Connect App，现在直接问 OpenClaw：

"今天的运动数据怎么样？"

它会调出今天的步数、骑行里程、睡眠时长、HRV 状态，用自然语言回复我。如果想看更详细的数据，它会生成一个可视化图表发过来。

数据来源是 Garmin Connect 的 API，通过 OpenClaw 的 Garmin Connect Skill 拉取。

场景五：NAS 文件管理

我经常需要在两台 NAS 之间整理文件、备份数据。以前要么 RDP 过去操作，要么用 Synology Drive，但都不够灵活。

现在用自然语言管理：

"帮我把上周的照片从飞牛备份到微联通"

OpenClaw 解析这条指令，连接两台 NAS 的 SFTP，执行文件操作，全程不需要我手动操作。两个 NAS 都是内网 IP，通过 Tailscale 打通，OpenClaw 通过 Tailscale 的地址访问，没有任何公网暴露。

踩坑经验：这些坑希望你别再踩

坑一：飞书消息加密

飞书自建应用支持消息加密，配置的时候如果填错了 Encryption Key，会出现"消息收不到但错误提示不明确"的问题。建议先把加密关掉，把基本功能调通，再开加密。

坑二：CalDAV 时区

CalDAV 服务器返回的时间戳有的是 UTC，有的是本地时间，OpenClaw 读取之后需要统一转换。如果不做时区转换，日历事件会显示错误的时间。这个问题排查了很久，建议在 CalDAV 服务端就统一用 UTC 存储。

坑三：QQ Bot 消息频率限制

QQ 频道机器人有严格的消息频率限制（每分钟 20 条左右），如果定时任务在同一分钟内触发多个消息推送，会被腾讯风控。解决方法是加一个消息队列，把同一分钟内要发的消息合并成一条。

坑四：OCR 识别错误传播

记账 OCR 偶尔会识别错金额，如果直接写入数据库就会出错。我的做法是：OCR 结果先发一条确认消息给用户，用户确认后再写入数据库。这个交互多了一步，但保证了数据准确性。

坑五：内网服务的 Tailscale 依赖

我的 NAS 和本地 OCR 服务都跑在内网，依赖 Tailscale 做内网穿透。有一次 Tailscale 断了，这些服务就全挂了。后来我在 OpenClaw 里加了健康检查脚本，检测到 Tailscale 断线就发告警。

效果总结：它真的帮我省时间了吗？

坦诚地说：省了，而且省了不少。

量化一下大概是这样：

  • 早间简报：每天省 15-20 分钟（不用刷多个 App）
  • 记账：每月省 30-40 分钟（OCR 直接识别，不需要手动输入）
  • 信息搜索：每次搜索省 5-10 分钟（多引擎并行 + 自动去重）
  • 文件管理：每月省 1-2 小时（用自然语言操作，不用 RDP 过去）

非量化的好处更明显：我不那么焦虑了。以前总担心漏掉什么消息、忘记查基金净值、忘了今天有什么安排，现在这些都被系统兜住了。

当然，OpenClaw 也不是银弹。它本质上是帮你把 AI 能力和各种外部服务串联起来的框架，真正的自动化逻辑还是需要自己写 Skill 配置。有些场景下 AI 的回复不够准确，需要人工纠错。

下一步想做什么

目前正在折腾的方向有两个：

第一个是多模态交互。 现在主要还是文字，图片和文件偶尔能处理，但不是特别顺。以后希望能发一张图就能分析、一段语音就能记账。

第二个是跨设备协同。 我的 Mac mini 是主力机，但有时候不在家，想远程触发一些操作。现在靠 Tailscale 能连上，但体验还不够无缝。希望以后能用手机上的 OpenClaw 客户端直接控制家里的服务。

写在最后

搭建这套系统花了我大概两三周的时间，中间踩了不少坑，但现在回头看非常值得。

如果你也是开发背景，或者对自动化有兴趣，建议先从一个小场景入手——比如先接一个消息通道，配一个定时任务，跑通之后再逐步扩展。不要一开始就想着把所有事情都自动化，那样容易烂尾。

工具是为目标服务的，OpenClaw 给了我一个足够灵活的基础设施，剩下的就是按照自己的需求去填充。

有问题欢迎留言交流，祝各位折腾愉快。
